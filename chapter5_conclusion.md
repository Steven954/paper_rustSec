# 5 总结与展望

本章对全文工作进行总结，归纳本研究的主要贡献与结论，客观分析研究的局限性，并对未来工作方向进行展望。

## 5.1 本文研究总结

本研究围绕 Rust 非正常控制流下的内存安全问题，从场景分类、评测框架构建到工具能力边界评估，形成了一套较为完整的研究体系。主要工作总结如下。

针对现有研究对 Rust panic 相关内存安全漏洞分类语义薄弱、缺乏系统性描述的问题，本研究提出了面向非正常控制流的场景化分类体系。该体系摒弃了按 UAF、double free 等“表象类型”划分的传统做法，转而从**触发点位置**、**资源状态**、**边界位置**三个维度刻画漏洞发生情境，将 Rust 中因 panic/unwind 引发的内存安全问题归纳为四类典型场景：**场景 S1（Drop 执行中途发生 panic）**：析构函数内部 panic 导致析构链条中断，引发半析构状态，典型后果包括 double free、UAF、资源泄漏及进程 abort。**场景 S2（未初始化/部分初始化状态下 panic 中断）**：MaybeUninit、set_len、扩容/拷贝过程中 panic，使不变量未建立完成即进入析构，典型后果包括未初始化读、释放未分配、UAF。**场景 S3（panic 跨 FFI 边界传播）**：Rust 与 C 交互时 panic 穿越 FFI 边界，ABI/栈展开不匹配导致所有权契约破坏，典型后果包括未定义行为、资源错配释放、崩溃或绕过清理。**场景 S4（自定义资源清理逻辑中 panic）**：手写 guard、allocator、事务性清理逻辑中 panic 打断手工恢复步骤，典型后果包括 double free、UAF、泄漏及状态不一致。该分类体系遵循语义贴合、可复现、可评测、贴近真实漏洞的设计原则，并建立了场景与漏洞类型的映射关系，为后续评测与工具改进提供了统一的语义基础。

在分类体系基础上，本研究构建了一套面向 Rust 静态分析工具的系统性评测框架。该框架以“场景”为基本分析单位，将工具能力置于明确的语义上下文中进行比较，包含基准数据集构建策略、评测流程与结果归一化、指标体系（基础指标、场景感知指标、成本指标）以及能力边界分析方法。基于该框架，本研究对 Rudra、MirChecker、FFIChecker 三款工具在 100 个非正常控制流相关用例上进行了系统评测。实验结果表明：场景特异性显著，FFIChecker 在 S3（跨 FFI panic）中 Recall 达 68.75%，Rudra 在 S1、S4 中领先，MirChecker 在 S2 的数值与路径分析类漏洞上有一定优势；能力边界清晰，各工具在非目标场景下 Recall 多为 0%，符合其设计定位；可运行性良好，除 MirChecker 在 1 个用例上失败外，其余工具均可稳定运行。上述结论为理解 Rust 非正常控制流下静态分析工具的能力边界提供了实证依据，也为工具选型与改进方向提供了参考。

本研究在评测过程中形成了可复用的最小触发样例模板与复现实践建议。评测管线通过自动化脚本实现，支持版本锁定、超时控制、结果归一化等环节，便于他人复现与扩展。针对各工具的结构性限制，本研究提出了具体改进建议：Rudra 需增加规则过滤以排除非安全告警、提升 Precision；MirChecker 可扩展对 unwind 路径与 Drop 调用顺序的建模；FFIChecker 可与 Rudra 或 MirChecker 组合使用，形成“FFI + 非 FFI”的互补覆盖。

## 5.2 本文研究的局限性与未来工作展望

本研究存在以下局限性，需要在解读结论时予以考虑。

**样例规模与分布**：评测数据集共 100 个用例，其中 S2 场景占 66%，S1 仅 4 个。S1 样例数较少，统计波动较大，对 S1 场景下工具能力的结论需谨慎推广。S2 用例来源多样，部分 unit-tests 本身并非设计用于触发 panic 相关漏洞，真值标签与工具能力存在一定错位。此外，样例主要来自工具自带测试与 trophy-case，对真实大型项目的代表性有限。

**真实漏洞复现困难**：部分真实漏洞（如 CVE）依赖特定 Rust 版本、依赖库版本及 panic 策略，在评测环境中难以稳定复现。本研究采用抽象化与最小化样例替代，虽能反映漏洞机制，但与真实漏洞的语义等价性仍需人工确认。此外，未深入工具源码细节，也未追求复现所有已知 CVE，对工具在真实漏洞上的表现评估存在局限。

**静态分析固有局限**：本研究聚焦静态分析工具，未涉及动态验证（如 Miri）、符号执行或形式化验证方法。静态分析在路径爆炸、跨 crate 分析、泛型与 trait 解析等方面存在固有局限，对复杂 panic 路径的建模能力有限。此外，评测仅覆盖三款工具，其他 Rust 安全分析工具（如 Kani、Prusti、MIRAI 等）未纳入对比，能力边界结论的普适性有待扩展。

基于本研究的成果与局限，未来工作可从以下方向展开：（1）**扩展更大规模基准**：扩充 S1 场景样例数量，从真实漏洞、标准库历史问题及社区报告中挖掘更多 Drop 中 panic 的案例；引入更多真实项目代码片段，建立持续更新的基准仓库。（2）**引入动态验证与符号执行**：将 Miri 等动态分析工具纳入评测框架，与静态分析形成互补；探索符号执行与约束求解方法，对关键 panic 路径进行可达性分析；结合静态与动态方法，构建混合分析管线。（3）**面向 unwind/Drop 的静态建模增强**：研究基于 MIR 的 unwind 路径显式建模方法；扩展 MIR 层对 Drop 调用顺序、析构链条中断、部分初始化状态传播的分析能力；探索将 FFIChecker 的 FFI 分析能力与 Rudra 的 unsafe 数据流分析相结合，形成跨场景的联合检测方案。（5）**工具链集成与工程实践**：将评测框架与 Cargo 构建流程深度集成；探索将分类体系中的检查规则融入 Clippy 或自定义 lint；总结最佳实践与编码规范，形成面向 Rust panic 安全的开发指南。

综上所述，本研究围绕 Rust 非正常控制流下的内存安全问题，建立了场景化分类体系、构建了基于场景的评测框架，并对 Rudra、MirChecker、FFIChecker 三款工具进行了系统评估，归纳了其能力边界与改进方向。尽管存在样例规模、复现难度、指标精度等方面的局限，本研究为填补“panic/展开语义”视角下的安全分析与工具评测空白提供了有益探索，也为后续 Rust 安全分析工具的改进与评测框架的完善奠定了基础。

# 第 3 章 基于状态碎化理论的异常控制流多维场景分类体系

## 3.1 引言

Rust 通过所有权（Ownership）、借用检查（Borrow Checker）以及生命周期推导（Lifetime Inference）在编译期建立了较强的内存安全约束。在线性控制流场景下，这些机制能够有效约束资源分配、转移与释放顺序，避免常见的悬垂引用和重复释放问题。然而，在系统编程实践中，为了实现底层内存操控、跨语言互操作和性能敏感优化，代码通常不可避免地进入 `unsafe` 区域。此时，语言层面的静态保证会被部分让渡给开发者的手工不变量维护。

当程序运行至异常路径（例如 `panic!`、断言失败、越界检查失败）时，控制流将从正常业务路径转移到解卷路径（Unwinding Path）。该转移并不等价于“失败即终止”的简单语义，而是伴随一系列隐式清理行为：编译器自动插入的 `Drop` 逻辑会按照生命周期规则回收资源。问题在于，异常触发时程序状态往往处于“尚未完成更新”的过渡区间，逻辑元数据与物理内存内容可能失配，进而导致清理链在错误前提下执行。这一过程通常先表现为局部状态不一致，随后才在解卷链中物化为可观测漏洞。

基于上述观察，本章目标并非继续按漏洞表象（如 UAF、double free）进行分类，而是从异常控制流与状态演化机制出发，构建根因驱动的场景分类框架。具体地，本章通过状态碎化理论给出异常路径上的形式化约束，结合 MIR 拓扑解释 `Drop` 与 `unwind` 的运行关系，再在触发位置边界、资源所处状态和隔离跨越度三维空间上定义 S1-S4 场景，并通过决策树实现可执行判定。该框架为后续样例标注、消融分析和能力边界评估提供统一语义基础。

## 3.2 基于状态碎化理论的方法设计

### 3.2.1 异常安全的形式化约束

在异常安全理论中，系统级代码的最低要求是满足“基本异常保证（Basic Guarantee）”：即使异常发生，也不应破坏关键资源不变量，尤其不应产生未定义行为级别的内存违规。对于 Rust `unsafe` 代码段，该要求可表述为：控制流被中断时，程序允许业务语义出现局部不连续，但必须维持资源释放与引用关系的安全闭包。

设 $\mathcal{I}_{own}(t,v)$ 表示线程 $t$ 对内存区域 $v$ 的所有权不变量，且该区域满足类型布局与别名约束。在正常控制流下有：

$$
\forall t,\ \text{NormalFlow}(t)\Rightarrow \mathcal{I}_{own}(t,v)
$$

其中 `NormalFlow` 表示指令按预期分支与调用顺序执行。该状态下，逻辑层元数据（例如容器 `len/capacity`）与物理层对象初始化状态保持一致。由此可见，异常安全的关键不是“程序是否 panic”，而是“panic 发生时不变量是否仍可被安全清理机制消费”。

### 3.2.2 状态碎化与三阶段演进

在容器扩容、对象搬移、裸指针写入等操作中，程序会短暂进入中间态：逻辑元数据可能先更新，而物理写入尚未完成。当 `panic` 在此窗口触发时，更新序列被中断，形成逻辑与物理的异步状态。本文称该现象为“状态碎化（State Fragmentation）”。

设逻辑状态转移函数为 $f_{meta}$，物理写入函数为 $g_{mem}$，则破裂状态集合可定义为：

$$
S_{broken}=\{\exists m\in Metadata,\exists p\in PhysicalMem\mid f_{meta}(m)\neq g_{mem}(p)\}
$$

在该定义下，异常控制流威胁可系统化为三阶段：

1. **触发期（Triggering Phase）**：异常条件被激活，控制权从业务路径切换到异常处理路径；
2. **碎化期（Fragmentation Phase）**：数据结构被固定在中间态，不变量局部失效，状态仍可继续传播；
3. **物化期（Vulnerability Materialization Phase）**：`Drop`/清理链在碎化态上执行，风险转化为具体漏洞后果。

该三阶段模型将“panic 触发”与“漏洞出现”区分开来，解释了为何部分异常在触发点并不直接崩溃，而是在后续清理节点才呈现严重后果。

### 3.2.3 MIR 拓扑与解卷机制建模

为解释上述三阶段如何在编译与运行时联动，需要将分析视角下沉至 MIR。MIR 中每个函数可表示为由基本块组成的控制流图，终止符（如 `Call`、`Assert`、`Drop`）决定后继边。启用 `panic=unwind` 时，相关终止符携带 `unwind` 分支，异常将沿清理边跳转至 Cleanup 块。

在图结构上，所有解卷可达清理块形成“倒置解卷树（Inverted Unwind Tree）”，其关键性质包括：

- **单向收敛性**：每个清理节点后继受限，最终收敛至函数边界或 `Resume`；
- **无环约束**：清理路径不形成环，保证解卷器可终止；
- **静态清理语义**：`Drop` 执行依赖编译期生成逻辑，不具备运行时状态自校验能力。

这意味着，当状态已碎化但 `Drop` 仍按静态顺序执行时，清理动作可能在错误前提下触发，从而产生“清理导致漏洞”的机制性风险。MIR 语义因此是连接“形式化碎化定义”与“工程漏洞后果”的关键桥梁。

### 3.2.4 三维场景空间与 S1-S4 定义

在上述理论基础上，本文以三维场景空间刻画异常控制流风险：

1. **触发位置边界（Trigger Position Boundary）**：异常发生在普通业务路径、析构路径还是跨语言接口路径；
2. **资源所处状态（Resource State）**：异常发生时资源处于初始化扩张态、活跃态或销毁态；
3. **隔离跨越度（Isolation Leap）**：控制流是否跨越 ABI 边界或并发隔离边界。

据此定义四类场景：

- **S1 析构异常重入（Destructor Exception Re-entry）**  
  异常发生在析构链内部，清理过程被再次中断，形成半析构对象。该状态在后续复用或再次释放时容易触发悬垂引用和双重释放问题。

- **S2 局部初始化逃逸（Partial Initialization Escape）**  
  资源初始化尚未完成即触发异常，逻辑元数据先行推进而物理对象未完全就绪，解卷阶段可能将未初始化区域误当作有效对象处理。

- **S3 跨语言边界盲区（Cross-language Boundary Blindness）**  
  异常跨越 FFI/ABI 边界传播，破坏跨语言栈帧与所有权契约，可能导致宿主侧资源泄漏、悬垂指针或未定义行为。

- **S4 并发状态孤儿化（Concurrent State Orphanization）**  
  并发更新在中间步骤被异常打断，锁释放后共享对象语义处于断裂态；后续线程虽能访问对象，但访问前提已不再成立。

相较于按漏洞表象分类，该定义方式能够直接对应异常路径中的状态演化机制，更适用于根因分析与场景化评测。

### 3.2.5 决策树判定与正交性论证

为保证分类互斥与覆盖完备，本文采用自上而下的判定树：

1. **ABI 判定节点**：若异常路径跨越 ABI 边界，直接判为 S3；
2. **并发判定节点**：若未跨 ABI，但处于并发独占上下文，判为 S4；
3. **生命周期节点**：其余样例按生命周期阶段区分，扩张期判 S2，消亡期判 S1。

该设计具有两方面优势。其一，判定顺序与风险边界层级一致，先处理跨域风险，再处理并发风险，最后处理单线程生命周期风险。其二，节点语义与证据抽取可一一映射，便于后续消融实验量化每个节点对分类性能的边际贡献。

从集合关系看，S3 对应跨域子空间，S4 对应并发子空间，S1/S2 对应单线程生命周期子空间。三类子空间两两以判定前提区分，因而分类满足互斥性；任一样例在决策树中均存在终止叶子，因而满足完备性。

### 3.2.6 场景到 CWE 的语义映射

为增强分类体系的工程可解释性，本文将 S1-S4 场景映射至常见 CWE 语义族：

- **S1**：双重释放、释放后使用（如 CWE-415、CWE-416）；
- **S2**：未初始化资源访问与未初始化指针使用（如 CWE-824、CWE-908）；
- **S3**：跨边界内存契约破坏与资源管理失配（如 CWE-119、CWE-404）；
- **S4**：并发同步不当与生命周期控制失效（如 CWE-362、CWE-664）。

该映射并非将场景简化为 CWE 标签，而是建立“机制场景-漏洞后果”的双向参照：一方面可将工程漏洞回溯到异常控制流机制，另一方面可将机制场景外化为标准化缺陷语义，便于跨工具、跨数据集对齐。

## 3.3 实验环境与设置

本节说明第三章分类实验的实施条件、数据组织、指标定义与对比方案。实验目标是验证第 3.2 节决策树在真实样例上的可判别性与可解释性，而非评估静态分析工具本身性能。

### 3.3.1 实验环境

本文实验运行于 Windows 11（版本 10.0.26200）平台，处理器架构为 AMD64。实验软件环境为 Python 3.14.2，Rust 工具链通过 `cargo` 调用，目标三元组为 `x86_64-pc-windows-gnullvm`，链接器为 LLVM-MinGW 的 `x86_64-w64-mingw32-clang`。本章分类方法属于规则驱动与结构化统计分析，不涉及神经网络训练过程，因此实验不依赖 GPU 加速。所有分类与消融流程均在 Python 脚本环境下实现并自动执行。

### 3.3.2 实验数据集

需要说明的是，本文“原始场景标签”并非外部人工标注金标准，而是依据第 3.2 节完整决策树规则对样例进行统一判定后形成的基线标签。基于该基线标签，本章进一步构建不同节点消融变体（`step1_only`、`step1_step2`、`step1_step3`）并生成对应预测标签，通过比较“变体预测-基线标签”的一致性，量化分类框架在节点缺失条件下的稳定性、边界误差结构与节点级贡献。

本章使用的样例集经一致性过滤后包含 100 个有效样例，场景分布如下：

| 场景 | 含义 | 样例数 | 占比 |
| :--- | :--- | ---: | ---: |
| S1 | 析构异常重入 | 4 | 4% |
| S2 | 局部初始化逃逸 | 66 | 66% |
| S3 | 跨越语言边界盲区 | 16 | 16% |
| S4 | 并发状态孤儿化 | 14 | 14% |

该分布呈显著长尾：S2 为多数类，S1 为极少数类。若仅报告准确率，模型可能通过“偏向多数类”获得表面高分，因此本章后续采用宏平均与分场景指标并行评估。

在样例表征层面，本文使用代码/接口证据驱动分类判定。每个样例抽取三类二值证据：

- `evidence_abi_boundary`：跨语言边界证据；
- `evidence_concurrency`：并发语义证据；
- `evidence_drop_phase`：析构阶段证据。

证据抽取全部来自样例可观测结构，不采用随机赋值或人工回填，确保判定链可追溯。

### 3.3.3 评估指标

考虑类别不均衡，本章采用以下指标：

$$
Accuracy=\frac{N_{correct}}{N}
$$

$$
Recall_s=\frac{TP_s}{TP_s+FN_s},\quad s\in\{S1,S2,S3,S4\}
$$

$$
F1_s=\frac{2TP_s}{2TP_s+FP_s+FN_s}
$$

$$
Macro\text{-}F1=\frac{1}{4}\sum_{s}F1_s
$$

其中，`Macro-F1` 用于平衡少数类贡献，混淆矩阵用于识别边界混淆对并支持误差归因。

### 3.3.4 参数设置与对比方案

本章为规则型分类实验，参数设置主要体现在“决策树节点启用方式”。

完整模型（`full`）按第 3.2 节决策树执行：

1. 若存在边界证据，判为 S3；
2. 否则若存在并发证据，判为 S4；
3. 否则若存在析构阶段证据，判为 S1；
4. 其余判为 S2。

为评估节点贡献，设置三种消融变体：

- `step1_only`：仅保留边界节点；
- `step1_step2`：保留边界与并发节点；
- `step1_step3`：保留边界与析构节点。

四组方案共同构成本章对比实验。

## 3.4 实验结果与分析

本节围绕“节点贡献-边界混淆-误判归因-可观测性补充”展开分析。

### 3.4.1 节点消融结果对比分析

四种方案总体结果见表 3-4。

| 方案 | Accuracy | Macro-F1 | Recall(S1) | Recall(S2) | Recall(S3) | Recall(S4) |
| :--- | ---: | ---: | ---: | ---: | ---: | ---: |
| step1_only | 0.8100 | 0.4605 | 0.0000 | 1.0000 | 0.9375 | 0.0000 |
| step1_step2 | 0.7300 | 0.5193 | 0.0000 | 0.8030 | 0.9375 | 0.3571 |
| step1_step3 | 0.8400 | 0.6661 | 1.0000 | 0.9848 | 0.9375 | 0.0000 |
| full | 0.7600 | 0.7248 | 1.0000 | 0.7879 | 0.9375 | 0.3571 |

从结果可得四点结论：

1. **边界节点对 S3 具有高辨识度**：`step1_only` 已能维持 S3 召回 0.9375；
2. **并发节点决定 S4 可见性**：去除并发节点时，S4 召回降为 0；
3. **析构节点决定 S1 可见性**：去除析构节点时，S1 召回降为 0；
4. **宏平均优先于准确率**：`step1_step3` 准确率更高但丢失 S4，`full` 在 `Macro-F1` 最优，更符合本任务目标。

该结果说明，决策树节点并非冗余叠加，而是分别对应不同场景的识别通道。

### 3.4.2 完整模型混淆矩阵分析

完整模型混淆矩阵见表 3-5。

| 真实场景 \\ 预测场景 | S1 | S2 | S3 | S4 |
| :--- | ---: | ---: | ---: | ---: |
| S1 | 4 | 0 | 0 | 0 |
| S2 | 1 | 52 | 0 | 13 |
| S3 | 1 | 0 | 15 | 0 |
| S4 | 0 | 9 | 0 | 5 |

该矩阵体现三类结构信息：

- **稳定区间**：S1 与 S3 的主对角占比高，说明析构语义和跨域语义相对清晰；
- **边界混淆区间**：S2 与 S4 存在双向混淆（13 与 9），说明并发语义与资源状态语义在边界样例中相互交叠；
- **少量跨域吸附**：S2、S3 各有 1 例被吸附到 S1，提示析构证据在个别样例中具有过强主导性。

因此，后续优化重点应放在 S2/S4 边界证据细化，而非全局模型重构。

### 3.4.3 典型误判样例归因

基于逐样例证据明细，误判可归纳为以下机制：

1. **析构证据吸附机制**  
   当样例同时具有析构线索与其他弱线索时，判定路径倾向落入 S1，造成少量跨类吸附。

2. **并发证据外溢机制**  
   某些 S2 样例存在与并发相关的接口或命名信号，触发 S4 判定，形成 `S2→S4`。

3. **并发证据缺失机制**  
   部分真实 S4 样例未显式暴露并发证据，按默认路径回落至 S2，形成 `S4→S2`。

上述误判具有稳定模式，说明当前分类体系具备可解释的“错误结构”，而非不可控随机误差。这为后续改进提供了明确方向：通过增强并发证据的精度与覆盖率，同时抑制析构证据外溢。

### 3.4.4 补充：可观测性对照分析（unwind/abort）

在补充对照中，`panic=unwind` 与 `panic=abort` 在触发计数、内存错误记录数和构建失败数上基本一致，主要差异体现在平均执行时延。该结果表明，在当前样例规模下，`panic` 策略切换更多影响运行代价而非宏观触发统计。

需要说明的是，补充对照样本规模较小且存在构建失败样例，因此该部分用于说明“可观测条件会影响实验表征充分性”，不作为主结论来源。

## 3.5 本章小节

本章完成了异常控制流场景分类体系从理论到实证的闭环构建。首先，基于异常安全约束提出状态碎化定义，并给出触发期-碎化期-物化期三阶段机制；其次，借助 MIR 解卷拓扑解释清理链在碎化态上的风险传播；再次，在三维场景空间中定义 S1-S4，并以决策树保证分类互斥与完备；最后，通过节点消融、混淆矩阵与误判归因验证了分类体系的可判别性与可解释性。

综合来看，本章输出了可复用的场景标签体系、可执行的判定流程与可诊断的边界误差模型。该成果为第四章开展分场景工具评测提供了统一语义基准，并为后续能力边界分析提供了可追溯解释框架。

# 第 5 章 基于场景分类的静态分析工具评测框架与能力边界分析

本章在前一章构建的非正常控制流场景分类体系基础上，进一步提出一套面向 Rust 静态分析工具的系统性评测框架。与传统基于漏洞类型或真实项目扫描规模的评测方式不同，本章的核心思想是以“场景”为基本分析单位，将工具能力置于明确的语义上下文之中进行比较，从而识别其在 panic 相关内存安全问题上的能力边界。为此，本章首先界定评测目标与研究问题，随后阐述基准数据集构建策略与评测流程设计，构建一套兼具通用性与场景感知能力的指标体系，并提出能力边界分析方法与有效性控制策略。

## 5.1 评测目标、问题定义与研究假设

本研究的评测目标并非简单比较不同工具的“检出数量”或“性能开销”，而是聚焦于工具在 S1–S4 四类非正常控制流场景下的检测能力、可运行性与能力边界。具体而言，评测关注以下三个层面：其一，工具是否能够在对应场景中识别资源不变量破坏所引发的内存安全风险；其二，工具是否能够在受限构建环境下成功运行并产生稳定输出；其三，当工具未能检出问题时，其能力边界位于何种语义层面，是控制流建模不足、数据流精度不足，还是工程层面约束所致。

围绕上述目标，本研究提出三个研究问题。第一，不同静态分析工具在不同场景下的检出能力是否存在显著差异，以及这种差异是否具有系统性。第二，导致漏报与误报的关键语义因素是什么，即工具在哪些语义维度上缺乏建模能力。第三，版本锁定、依赖库差异、panic 策略选择等构建约束对评测结论的影响程度如何，以及如何通过工程控制手段提升评测结果的可复现性。

基于对现有工具设计理念的分析，可以提出如下研究假设：不同工具在特定场景中存在结构性盲区。例如，以不安全数据流为核心的工具可能更易发现 S2 场景中的不变量破坏问题，而对 S1 中 Drop 链条中断的路径敏感性不足；侧重内存释放模式匹配的工具可能对 double free 表现敏感，但对 FFI 边界异常传播缺乏建模能力。通过场景化评测，可以验证这些假设，并对工具能力进行结构化刻画。

## 5.2 基准数据集构建策略

评测框架的有效性首先取决于基准数据集的科学性。由于 panic 触发的内存安全漏洞具有高度语义依赖性，单纯随机收集代码片段难以构建高质量评测样本。因此，本研究采用“真实漏洞抽象化 + 最小可复现样例构造”的双轨策略构建数据集。

数据集的来源包括公开漏洞数据库中披露的 Rust 安全问题、相关论文中报告的漏洞案例，以及开源仓库中与 panic 安全相关的修复提交。对于每一个候选样例，均需满足三个纳入标准：其一，代码路径中存在可触发的非正常控制流，包括显式 panic、隐式断言失败或异常传播；其二，能够明确定位资源不变量被破坏的位置与时刻，从而定义“真值标签”；其三，在锁定版本与构建环境后能够稳定复现。

为了降低版本漂移对评测结果的干扰，每个样例均附带完整的构建描述文件，包括 rust-toolchain.toml、Cargo.lock、编译 profile 配置、feature flags 以及 panic 策略设置。在必要情况下，对编译选项进行显式声明，例如区分 panic=unwind 与 panic=abort 的行为差异。通过锁定依赖图与工具链版本，可以最大限度降低环境差异对评测结论的影响。

在环境固化方面，本研究采用容器化与脚本化相结合的方式。每一组工具与样例运行环境均封装为独立容器镜像或自动化构建脚本，从而保证不同实验批次之间的一致性。所有样例均以最小工程形式组织，使得构建过程不依赖外部仓库网络访问，避免由于依赖更新导致不可重复。

## 5.3 评测流程与结果归一化

在完成样例构建后，本研究设计了一条标准化评测管线。首先在锁定版本的环境中构建样例工程，其次运行目标静态分析工具，随后收集工具输出日志，并将其转换为统一结构的数据格式。在此基础上进行人工确认与统计分析。

由于不同工具输出格式差异较大，有的以编译错误形式呈现，有的以 lint 警告形式输出，有的仅给出规则编号，因此需要构建统一的归一化 schema。该 schema 至少包含工具名称、样例编号、对应场景类别、告警位置、告警类型、规则标识符、置信度信息以及运行状态等字段。对于未能成功运行的工具，应明确记录其失败原因，例如构建失败、版本不兼容或运行时崩溃。

人工确认是评测流程中的关键环节。为避免主观偏差，本研究制定统一的判定标准：当工具告警准确定位资源不变量破坏路径且语义相关时，判定为检出；当告警与样例无关或仅提示泛化风险而无法关联到真实漏洞时，判定为误报；当样例中存在真实漏洞但工具未给出相关提示时，判定为漏报。通过双人复核机制提升标注一致性，并在存在争议时记录判定依据。

## 5.4 指标体系

### 5.4.1 基础指标

基础指标用于刻画工具在传统信息检索意义上的性能表现。首先是 Recall，即在某一场景中被正确检出的样例比例。该指标直接反映工具在该场景下的覆盖能力。其次是 Precision，用于衡量工具告警的准确性，避免以高误报率换取表面上的高检出率。漏报率 FNR 则与 Recall 互补，用于强调工具在真实风险场景中的遗漏程度。

除了检测性能外，可运行率（Executability）同样重要。某些工具由于版本依赖或构建约束，可能在部分样例上无法成功运行。可运行率用于统计在全部样例中成功产生有效输出的比例，从工程可用性角度反映工具成熟度。

### 5.4.2 场景感知指标

为体现本研究场景分类框架的价值，需要构建场景感知指标。场景覆盖率用于统计工具在 S1–S4 四类场景中至少成功检出一次的比例，反映其语义广度。Drop 感知度用于衡量工具是否显式建模析构路径与 unwind 行为，若工具仅在正常路径下分析资源释放，则其在 S1 场景中的表现通常较弱。边界感知度则关注工具是否识别 safe/unsafe 边界、跨 crate 或跨 FFI 边界的语义差异，尤其是在 S3 场景中表现尤为关键。

通过这些指标，可以从语义维度而非单一检出率维度刻画工具能力结构。

### 5.4.3 成本指标

除了检测能力，运行成本与配置复杂度同样构成工具能力评价的一部分。运行时间与资源占用反映工具在大规模项目中的可扩展性；配置成本则包括依赖安装复杂度、版本锁定难度、是否需要自定义编译器等工程因素。在实际应用中，高配置成本往往成为工具难以推广的重要原因，因此需要在评测中予以记录。

## 5.5 能力边界分析方法

在获得量化指标后，本研究进一步通过边界归因方法解释差异来源。边界归因从四个维度展开：控制流建模能力、数据流精度、语义建模深度以及工程约束。若某工具在 S1 场景中持续漏报，而在 S2 中表现良好，则可能说明其对 unwind 控制流建模不足；若在别名共享场景中漏报，则可能源于数据流分析精度不足；若在 FFI 场景下无任何告警，则可能未建模跨语言边界语义。

最终输出形式包括能力矩阵与雷达图，通过可视化方式展示各工具在不同维度上的能力分布。通过这种结构化分析，可以明确指出每个工具的优势领域与盲区，为未来改进方向提供依据。

## 5.6 有效性威胁与缓解

在构建评测框架过程中，需考虑多种有效性威胁。构念有效性方面，若样例未能真实反映 panic 语义下的不变量破坏，则评测结论将失去意义。内部有效性方面，若人工标注存在偏差或环境配置不一致，则可能影响统计结果。外部有效性方面，样例规模与复杂度可能限制结论对真实大型项目的推广能力。

为缓解上述威胁，本研究采用版本锁定与环境容器化策略保证构建一致性，通过脚本化运行流程减少人为操作差异，并引入双人复核机制提升标注一致性。此外，对所有实验数据进行公开存档，以支持他人复现。

## 5.7 本章小结

本章在前述场景分类体系基础上，构建了一套面向 Rust 非正常控制流内存安全问题的静态分析工具评测框架。该框架从评测目标界定、数据集构建、流程归一化、指标设计到能力边界分析方法形成完整闭环。通过引入场景感知指标与能力边界归因维度，本研究突破了传统基于漏洞数量比较的评测方式，使不同工具在 panic 语义下的能力差异得以结构化呈现。该框架为下一章的实证分析奠定方法论基础，并为后续 Rust 安全分析工具的改进方向提供理论支撑。

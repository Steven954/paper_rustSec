# 第 7 章 总结与展望

本章对全文工作进行总结，归纳本研究的主要贡献与结论，客观分析研究的局限性，并对未来工作方向进行展望。

## 7.1 工作总结

本研究围绕 Rust 非正常控制流下的内存安全问题，从场景分类、评测框架构建到工具能力边界评估，形成了一套较为完整的研究体系。主要工作总结如下。

### 7.1.1 非正常控制流场景分类体系

针对现有研究对 Rust panic 相关内存安全漏洞分类语义薄弱、缺乏系统性描述的问题，本研究提出了面向非正常控制流的场景化分类体系。该体系摒弃了按 UAF、double free 等“表象类型”划分的传统做法，转而从触发点位置、资源状态、边界位置三个维度刻画漏洞发生情境，将 Rust 中因 panic/unwind 引发的内存安全问题归纳为四类典型场景：

- 场景 S1（Drop 执行中途发生 panic）：析构函数内部 panic 导致析构链条中断，引发半析构状态，典型后果包括 double free、UAF、资源泄漏及进程 abort。
- 场景 S2（未初始化/部分初始化状态下 panic 中断）：MaybeUninit、set_len、扩容/拷贝过程中 panic，使不变量未建立完成即进入析构，典型后果包括未初始化读、释放未分配、UAF。
- 场景 S3（panic 跨 FFI 边界传播）：Rust 与 C 交互时 panic 穿越 FFI 边界，ABI/栈展开不匹配导致所有权契约破坏，典型后果包括未定义行为、资源错配释放、崩溃或绕过清理。
- 场景 S4（自定义资源清理逻辑中 panic）：手写 guard、allocator、事务性清理逻辑中 panic 打断手工恢复步骤，典型后果包括 double free、UAF、泄漏及状态不一致。

该分类体系遵循语义贴合、可复现、可评测、贴近真实漏洞的设计原则，并建立了场景与漏洞类型（UAF、double free、uninit、leak、FFI-UB）的映射关系，为后续评测与工具改进提供了统一的语义基础。

### 7.1.2 基于场景分类的评测框架与能力边界结论

在分类体系基础上，本研究构建了一套面向 Rust 静态分析工具的系统性评测框架。该框架以“场景”为基本分析单位，将工具能力置于明确的语义上下文中进行比较，包含以下核心组成部分：

1. 基准数据集构建策略：采用“真实漏洞抽象化 + 最小可复现样例构造”的双轨策略，通过 rust-toolchain.toml、Cargo.lock、panic 策略等版本锁定手段保障可复现性。
2. 评测流程与结果归一化：设计标准化评测管线（构建 → 运行工具 → 收集输出 → 归一化 → 人工确认 → 统计），建立统一的 schema 将不同工具输出转换为可比较的结构化数据。
3. 指标体系：涵盖基础指标（Recall、Precision、FNR、Executability）、场景感知指标（场景覆盖率、Drop 感知度、边界感知度）及成本指标（运行时间、配置复杂度）。
4. 能力边界分析方法：从控制流建模、数据流精度、语义建模、工程约束四个维度进行边界归因，输出工具能力矩阵与图谱。

基于该框架，本研究对 Rudra、MirChecker、FFIChecker 三款工具在 100 个非正常控制流相关用例上进行了系统评测。实验结果表明：

- 场景特异性显著：FFIChecker 在 S3（跨 FFI panic）中 Recall 达 68.75%，Rudra 在 S1、S4 中领先，MirChecker 在 S2 的数值与路径分析类漏洞上有一定优势。
- 能力边界清晰：各工具在非目标场景下 Recall 多为 0%，符合其设计定位；Rudra 在 S3 上无实质能力，MirChecker 与 FFIChecker 对 S4 无建模。
- 可运行性良好：除 MirChecker 在 1 个用例上失败外，其余工具均可稳定运行；FFIChecker 平均耗时约为 Rudra 的 5 倍。

上述结论为理解 Rust 非正常控制流下静态分析工具的能力边界提供了实证依据，也为工具选型与改进方向提供了参考。

### 7.1.3 可复现流程与实践建议

本研究在评测过程中形成了可复用的最小触发样例模板与复现实践建议。评测管线通过 `eval` 目录下的自动化脚本实现，支持版本锁定、超时控制、结果归一化等环节，便于他人复现与扩展。针对各工具的结构性限制，本研究提出了具体改进建议：Rudra 需增加规则过滤以排除非安全告警、提升 Precision；MirChecker 可扩展对 unwind 路径与 Drop 调用顺序的建模；FFIChecker 可与 Rudra 或 MirChecker 组合使用，形成“FFI + 非 FFI”的互补覆盖。

## 7.2 局限性

本研究存在以下局限性，需要在解读结论时予以考虑。

### 7.2.1 样例规模与分布

评测数据集共 100 个用例，其中 S2 场景占 66%，S1 仅 4 个。S1 样例数较少，与真实 Drop 中 panic 的样例相对稀缺有关，统计波动较大，对 S1 场景下工具能力的结论需谨慎推广。S2 用例来源多样，部分 unit-tests 本身并非设计用于触发 panic 相关漏洞，真值标签与工具能力存在一定错位。此外，样例主要来自工具自带测试与 trophy-case，对真实大型项目的代表性有限。

### 7.2.2 真实漏洞复现困难

部分真实漏洞（如 CVE）依赖特定 Rust 版本、依赖库版本及 panic 策略，在评测环境中难以稳定复现。本研究采用抽象化与最小化样例替代，虽能反映漏洞机制，但与真实漏洞的语义等价性仍需人工确认。此外，未深入工具源码细节，也未追求复现所有已知 CVE，对工具在真实漏洞上的表现评估存在局限。

### 7.2.3 评测指标与解析精度

当前 Recall 计算未严格区分真阳性与误报。Rudra 的解析器将 `error`、`warning` 等泛化关键词视为检出，导致大量非安全告警（如 `cargo metadata` 失败、`unused variable`）被计入检出，Recall 存在虚高。若进行人工复核，Rudra 的真实 Recall 应显著低于报告值。Precision 与 F1 的完整计算依赖人工标注，本研究尚未完成全量人工确认，因此对工具准确率的刻画仍不充分。

### 7.2.4 静态分析固有局限

本研究聚焦静态分析工具，未涉及动态验证（如 Miri）、符号执行或形式化验证方法。静态分析在路径爆炸、跨 crate 分析、泛型与 trait 解析等方面存在固有局限，对复杂 panic 路径的建模能力有限。此外，评测仅覆盖三款工具，其他 Rust 安全分析工具（如 Kani、Prusti、MIRAI 等）未纳入对比，能力边界结论的普适性有待扩展。

## 7.3 未来工作

基于本研究的成果与局限，未来工作可从以下方向展开。

### 7.3.1 扩展更大规模基准

扩充 S1 场景样例数量，从真实漏洞、标准库历史问题及社区报告中挖掘更多 Drop 中 panic 的案例。引入更多真实项目代码片段，提升基准对实际工程场景的代表性。建立持续更新的基准仓库，随 Rust 生态演进与漏洞披露不断补充新样例，形成长期可用的评测资源。

### 7.3.2 引入动态验证与符号执行

将 Miri 等动态分析工具纳入评测框架，与静态分析形成互补。Miri 可在运行时执行字节码并捕捉未定义行为，对 panic 路径的验证具有独特价值。探索符号执行与约束求解方法，对关键 panic 路径进行可达性分析，验证静态分析漏报的根因。结合静态与动态方法，构建混合分析管线，提升漏洞检出率与置信度。

### 7.3.3 面向 unwind/Drop 的静态建模增强

针对现有工具在 S1、S4 场景下的盲区，研究基于 MIR 的 unwind 路径显式建模方法。扩展 MIR 层对 Drop 调用顺序、析构链条中断、部分初始化状态传播的分析能力，设计专门针对 panic safety 的检测规则。探索将 FFIChecker 的 FFI 分析能力与 Rudra 的 unsafe 数据流分析相结合，形成跨场景的联合检测方案。

### 7.3.4 人工标注与指标完善

完成全量样例的人工标注，区分真阳性与误报，计算 Precision、F1 等完整指标。建立双人复核与争议解决机制，提升标注一致性。在此基础上，细化场景感知指标（如 Drop 感知度、边界感知度的量化定义），使能力边界分析更加精确。

### 7.3.5 工具链集成与工程实践

将评测框架与 Cargo 构建流程深度集成，支持开发者在本机一键运行多工具对比。探索将分类体系中的检查规则融入 Clippy 或自定义 lint，为开发者提供即时的 panic 安全提示。总结最佳实践与编码规范，形成面向 Rust panic 安全的开发指南，推动研究成果在工业界的落地应用。

---

综上所述，本研究围绕 Rust 非正常控制流下的内存安全问题，建立了场景化分类体系、构建了基于场景的评测框架，并对 Rudra、MirChecker、FFIChecker 三款工具进行了系统评估，归纳了其能力边界与改进方向。尽管存在样例规模、复现难度、指标精度等方面的局限，本研究为填补“panic/展开语义”视角下的安全分析与工具评测空白提供了有益探索，也为后续 Rust 安全分析工具的改进与评测框架的完善奠定了基础。

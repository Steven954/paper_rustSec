# 第 4 章 基于场景分类的静态分析评测框架

## 4.1 引言

随着 Rust 语言在系统级开发中的渗透，由于非正常控制流（特别是 Panic 异常机制）所引发的内存泄漏、双重释放以及状态不变式破损问题，已成为制约复杂系统高可用性的重要瓶颈。然而，现有的大多数代码检测工具评测往往停留在比对“发现已知漏洞的绝对数量”或“单一项目扫描开销”上，缺乏将工具能力置于清晰的语义上下文中进行剖析的维度。本章在前文构建的 Rust 异常控制流安全场景分类体系（S1-S4）基础之上，旨在提出并实现一套具有场景语义感知能力的静态分析评测框架。

围绕该评测目标，本章提出以下三个核心研究假设与问题：第一，由于底层分析理论的根本差异，不同分析工具在面对各类异常控制流场景时，其检出能力必然存在显著且系统性的“领域特异性”；第二，导致工具出现系统性漏报或海量误报的关键因素，深埋于其对中间表示中 Unwind 路径、跨语言边界（FFI）以及底层数学约束的建模能力之中；第三，Rust 自身繁杂的版本迭代以及强依赖特征，将直接构成制约工具工程可用性的重要威胁。通过引入场景化的实验设定与归因维度，本章对 Rudra、MirChecker 和 FFIChecker 三款工具进行评测，量化其能力表现，并归纳现有技术的能力边界与可能的演进方向。

## 4.2 静态分析评测框架与分析方法设计

本节阐述评测框架的设计方法，该框架通过体系化的用例构造、标准化的运行管线以及多维度的指标设定，支持对分析工具的综合效能进行度量。

### 4.2.1 基准数据集构建策略

评测框架的有效性首先取决于基准数据集的科学性与代表性。由于 Panic 触发的内存安全漏洞具有较强的控制流依赖性，随机提取的代码片段无法构成高质量的评测样本。为此，本研究采用“真实漏洞抽象化 + 最小可复现样例构造”的双轨策略，从开源漏洞库（如 CVE、RustSec）以及顶会论文报告中提取并重构测试用例。候选用例的筛选必须严格满足三个条件：其一，代码路径中存在显式的 Panic 调用、隐式断言失败或跨界异常传播机制；其二，能够精确锁定导致资源不变式被破坏的物理代码行，以便确立“真值标签（Ground Truth）”；其三，在特定的构建环境下能稳定复现漏洞。为阻断 Rust 生态中常见的版本漂移干扰，每一测试样例均被强绑定了完整的工程配置（包含 rust-toolchain.toml 及 Cargo.lock），并对编译时的 Panic 行为（panic="unwind" 或 abort）进行了显式隔离与声明。

### 4.2.2 评测运行管线与结果归一化

为保障多工具横向对比的工程客观性，本研究开发了一条全自动化的评测管线。由于不同分析工具（如 Linter 风格的报错、基于规则编号的隐晦输出等）产生的数据异构性极大，管线的核心任务是对异构日志进行提取与清洗。该管线按以下顺序执行：首先在隔离的容器内拉起工具分析特定项目，收集标准输出日志，随后将其转换为统一的归一化 Schema。该 Schema 被定义为包含：工具名称、对应场景标识（S1-S4）、告警发生位置、内置规则编号及运行状态。在结果判定上引入双人复核机制：当且仅当告警准确定位于导致不变式破裂的关键异常路径时，记为有效检出（True Positive）；若提示过于宽泛或根本不涉及目标漏洞，则一律划归为误报（False Positive）。

### 4.2.3 评测指标体系与能力边界归因方法

在量化衡量方面，除了传统的召回率（Recall）、精确率（Precision）以及体现工具稳定性的可运行率（Executability）等基础统计学指标外，本框架引入了旨在刻画语义能力的场景感知指标与边界归因方法：

场景覆盖度与感知度：统计工具是否能够横跨 S1 乃至 S4 获取至少一次成功检出；同时考察工具对底层异常路径（Drop-awareness）及跨语言边界（Boundary-awareness）的解析灵敏度。

能力边界归因矩阵：当工具在某类场景中检出率很低时，将从四大底层技术维度展开归因：(1) 控制流建模深度（是否完整追踪了 MIR 中的 Unwind 分支）；(2) 数据流与约束精度（是否使用了强符号求解应对复杂指针）；(3) 跨域语义范围（对 C/C++ 外部符号是否具备黑盒穿透能力）；(4) 工程设计依赖（对 AST 宏展开或异步底层的兼容性）。

## 4.3 实验设置与结果分析

本节详细介绍实验的具体配置，并展示三款参评工具在上述评测框架下对于 100 个非正常控制流测试用例的综合表现及深层原因分析。

### 4.3.1 实验设置

实验环境与对象配置：所有实验统一在 Windows 10 操作系统下运行，核心执行流由 Python 3 调度引擎接管，并依托 Docker 容器保证编译环境的纯净与依赖隔离。为应对混合语言编译的极端开销，单次运行上限超时被设置为 1200 秒。参评的静态分析工具包括：

- Rudra：基于抽象语法树与 MIR 数据流，主攻 Unsafe 内存模式与 Panic 滥用的大规模启发式工具。
- MirChecker：基于底层抽象解释与符号执行，擅长捕获算术溢出与越界导致异常的深层推理框架。
- FFIChecker：专注 FFI 跨语言交互安全，专职捕获异常逃逸与外部内存损坏的边界检测工具。

数据集概览如表 4-1 所示，总计 100 个有效用例被严格划分入四大非正常控制流场景。其中 S2 场景（未完全初始化对象遭遇 Panic 导致后续释放出错）占比达 66%，这与该模式在工程实践中较易引发安全问题的现状相符。S1 占比较小（4%），反映出标准库 Drop 内部直接嵌套 Panic 极易引发进程立即 Abort，此类样本相对罕见。

一、从数据反差到方法归因：启发式与形式化并非“谁更先进”，而是“谁更匹配场景”：若仅看总体指标，Rudra（47%）明显高于 MirChecker（14%），两者相差 33 个百分点。该差距并不意味着符号执行在理论上失效，而是揭示了 Panic 驱动异常控制流这一问题域的工程事实：当漏洞触发路径高度依赖编译器隐式生成的清理与解卷绕逻辑时，优先利用结构模式进行“高召回筛查”的启发式策略，往往比依赖路径完备枚举的形式化策略更能稳定落地。换言之，本实验的核心结论不是“启发式优于形式化”，而是“在隐式异常控制流主导的场景中，启发式具有更高的工程有效性上限”。

二、启发式模式在隐式控制流中的优势（S1/S2）：S1（析构异常）与 S2（半初始化异常）是典型的隐式控制流密集场景，程序真实执行轨迹往往由编译器插入的 Cleanup Blocks、析构顺序与 Unwind 跳转共同决定。Rudra 在这两类场景上保持较高检出（S1: 75.00%，S2: 51.52%），说明其基于结构模式的风险预测能够较好贴合“资源状态突变 + 异常中断”这一高频失效形态。与之对比，MirChecker 在 S1/S2 的召回明显偏低（S1: 25.00%，S2: 19.70%），其主要瓶颈并非规则缺失，而是形式化路径枚举在隐式分支激增后更容易受到状态爆炸牵制。由此可推导出一个更具指导意义的启发式思考：对于由编译器隐式生成且规模快速膨胀的异常路径，应优先以“结构先验 + 风险模式”完成第一阶段覆盖，再将符号执行聚焦于少量高置信候选路径做精化验证。

三、通用工具的跨边界语义盲区（S3）：S3（跨语言边界）给出了另一组更具冲击力的数据反差。对 Rust 单语言语义优化的通用工具在该场景基本失效：Rudra 与 MirChecker 的有效检出可视为 0%，而专属跨语言工具 FFIChecker 达到 68.75%。这一结果指向同一个能力边界：主流静态分析工具普遍存在“单体语义近视（Monolithic Semantic Myopia）”，即在语言内控制流与内存模型上表现良好，但对 `extern "C"` 协议、跨边界异常传播约束和外部符号语义缺乏统一建模。该发现意味着，未来安全工具若继续局限于单语言闭环优化，即便提升局部求解精度，也难以覆盖真实系统中的跨边界高风险链路；具备跨语言协议级联分析能力将是下一代工具的必要条件，而非可选增强项。

四、能力边界的综合结论与演进方向：综合 S1-S4 可见，Rudra 代表的启发式方法在隐式异常控制流场景具备更强的工程鲁棒性，MirChecker 代表的形式化方法在局部约束验证上仍有不可替代价值，FFIChecker 则证明“跨语言语义建模”能够显著抬升边界场景检出能力。由此，本研究建议后续工具体系沿“分层协同”演进：以启发式扫描承担宽覆盖，以形式化分析承担高置信确认，以跨语言协议分析补足语义断层，形成面向真实工业代码的三段式联合检测链路。

## 4.4 有效性威胁与缓解

任何软件分析评测均不可避免地引入有效性威胁。本章在框架设计与实施过程中采取了多重缓解措施：

构念与内部有效性（Construct & Internal Validity）：为防止自行合成的样例脱离生产实际导致结论失真，本数据集中近半数案例强制来源于工业界中已备案的 CVE/RustSec 历史通报。此外，评测管线中采用的严苛编译器版本锁定、Docker 镜像固化以及双人盲审冲突仲裁机制，以降低因操作环境差异或主观解读带来的偏差。

外部有效性（External Validity）：100 个微型或中等规模的提纯代码集可能无法完全复现百万级庞大工程（如 Linux 内核驱动）中跨模块调用的浩瀚状态空间。未来的延伸研究预留了向分布式测试集群无缝扩容的接口，并公开了所有中间运行态日志，以供安全研究社区对真实庞大语料库进行二次复现与校准。

## 4.5 本章小结

本章提出并实现了一套面向 Rust 非正常控制流（Panic 异常）内存安全问题的静态分析评测框架。通过对测试用例选取、容器化运行环境及多源结果的归一化，构建了可复现的评价流程，并引入场景感知度与边界归因分析等度量手段，以补充仅凭检出率评判工具的不足。基于该框架，本章对 Rudra、MirChecker、FFIChecker 三款工具进行了评测。结果表明，各工具呈现出较明显的领域分工与局限：Rudra 在 S2、S4 场景（半初始化与自定义清理）上表现较好；FFIChecker 在跨语言边界与 Panic 逃逸（S3）方面具有优势；MirChecker 在约束求解与局部验证上精度较高，但对隐式异常路径的建模不足。上述发现为理解现有工具的能力边界提供了实证依据，也为后续研发混合分析架构、完善 Rust 系统安全工具链提供了参考。
